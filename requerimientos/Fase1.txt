# System Specification: Custom Sales & Order Management System (Multi-tenant)

## Context & Tech Stack
* **Framework:** ASP.NET Core 10 MVC (Clean/Basic layered architecture).
* **Hosting:** Azure Container Apps / Azure Container Registry.
* **Database:** SQL Server utilizing Entity Framework Core (Code-First Migrations).
* **Development Methodology:** Spec-Driven Development (SDD) via Claude Code.
* **Language & Standards:** UI language is Spanish (due to local Costa Rican context/currency), but **100% of the source code, variables, database fields, comments, and logs must be in English**.
* **Multi-tenancy:** Data isolation by Organization (`OrganizationId`). Every process, query, and entity must filter by the active organization stored in the user session, except shared entities.

## UI/UX Framework
* **Template:** AdminLTE-3.0.5 (Bootstrap & jQuery).
* **Behavior:** Full AJAX interaction for CRUDs and state changes. A global loading spinner must block the UI during asynchronous server requests.
* **Session Management:** No authentication mechanism is required for now. Provide a global UI dropdown selector in the layout to switch the active `OrganizationId` stored in `HttpContext.Session`.

## Domain Models & Database Schema

### 1. Configuration
* General system parameters (Default Tax Percentage, Default Exchange Rate, Local Currency ID).

### 2. Currency
* Catalog of currencies used for pricing and orders.

### 3. Organization
* Tenant catalog to isolate transactional data.

### 4. ProductType
* Catalog for product categorization. Includes business logic metadata to calculate manual/automatic shipping estimates and service fees.

### 5. Supplier
* Catalog of vendors/websites where purchases are made (e.g., Amazon, Shein, Generic).

### 6. Customer
* Client registry. **Note:** Customers are shared globally across organizations.

### 7. Order
* Header for purchase orders. All fields must remain editable by the user, even calculated ones.
* **Fields:**
    * `Id` (GUID/INT)
    * `OrganizationId` (FK)
    * `NameOfOrder` (Open text)
    * `SupplierId` (FK)
    * `ExchangeRate` (Decimal, defaults from Configuration)
    * `TaxPercentage` (Decimal, defaults from Configuration)
    * `ShippingAmountIntern` (Decimal, manual input for import costs)
    * `ShippingAmountToCR` (Decimal, sum of individual item shipping costs)
    * `DiscountAmount` (Decimal, manual discount)
    * `TotalWithoutTaxes` (Decimal, sum of `RealPrice` from linked OrderItems)
    * `TaxesAmount` (Decimal, calculated tax sum based on order rules)
    * `TotalToPayToSupplier` (Decimal, calculated as: `TotalWithoutTaxes + TaxesAmount + ShippingAmountIntern - DiscountAmount`)
    * `TotalOfTheOrder` (Decimal, calculated as: `TotalToPayToSupplier + ShippingAmountToCR`)
    * `EstimatedProfitInLocal` (Decimal, calculated as: Sum of `AgreedPriceInLocal` from items minus `TotalOfTheOrder` converted to local currency)
    * `Status` (Enum: `Pending`, `Active`, `Delivering`, `Delivered`, `Completed`, `Voided`). Default view filters for `Pending` and `Active`.
    * *Rules:* If `Status` is `Pending`, all fields are editable. For any other status, fields become read-only (except via explicit transactional transitions).

### 8. OrderItem
* Line items belonging to an Order.
* **Fields:**
    * `Id`, `OrderId` (FK)
    * `CustomerId` (FK, different customers can exist within the same order)
    * `ProductDescription` (Text), `ProductLink` (URL), `ProductSourceCode` (Text), `ProductImage` (Blob/URL)
    * `ProductTypeId` (FK)
    * `ListPrice` (Base retail price without taxes or discounts)
    * `ListPriceTax` (Calculated using `TaxPercentage` over `ListPrice`)
    * `RealPrice` (Actual price paid to supplier excluding tax)
    * `EstimateShipping` (Calculated based on `ProductType` rules)
    * `ServiceFeeInLocal` (Estimated profit margin calculated via `ProductType`)
    * `AgreedPriceInLocal` (Final selling price. Calculated as: `(ListPrice + ListPriceTax) converted to Local Currency + ServiceFeeInLocal`. Must be editable, but UI must always display the original calculated formula value for reference).

### 9. Transaction
* An immutable ledger system to track all financial movements and enable state reconstruction at any given point in time. Automatically generated via system events, but manual overrides are permitted.
* **Fields:**
    * `Id` (GUID)
    * `OrganizationId` (FK, for isolation)
    * `SourceId` (Nullable FK, points to `OrderItemId` if applicable)
    * `Source` (Enum: `TransactionSource`)
    * `CustomerId` (Nullable FK)
    * `TransactionType` (Enum: `TransactionType` -> e.g., `Charge`, `Void`, `Payment`)
    * `TransactionDescription` (Autogenerated string based on source and type)
    * `TransactionDate` (DateTime, Costa Rica Local Time)
    * `Status` (Enum: `TransactionStatus`)
    * `CurrencyId` (FK, defaults to Local Currency)

## Core Business Rules & Ledger Automation
Transactions must be written to the ledger automatically based on the following `Order.Status` lifecycle changes:
1.  **Order Transition to `Active`:** Insert a `Transaction` for **each** `OrderItem` with value = `AgreedPriceInLocal` and type = `Charge`.
2.  **Order Transition to `Voided` (from any status other than Pending):** Insert a `Transaction` for **each** `OrderItem` with value = `AgreedPriceInLocal` and type = `Void`.
3.  **Payment Received:** When a payment is manually registered via the UI, insert a `Transaction` for **each** corresponding `OrderItem` with value = `AgreedPriceInLocal` and type = `Payment`.

## Required Views / Views Specifications

1.  **Standard Catalogs (CRUD):** Simple AJAX-driven lists to Add, Edit, and Delete rows for `Configuration`, `Currency`, `ProductType`, `Supplier`, and `Customer`.
2.  **Order Management Dashboard:**
    * Displays an overview list filtering `Pending` and `Active` orders by default.
    * Allows creating new orders, changing order status following the state machine workflow, and editing `Pending` orders.
3.  **Order Items Editor:**
    * Deep-dive view for a single order showing its `OrderItem` grid.
    * Full CRUD capabilities enabled **only** if the parent order status is `Pending`.
4.  **Payment Registry:**
    * UI screen to select a `Customer` and input an amount to pay.
    * Submitting records a new `Payment` transaction.
    * Must display the selected customer's total outstanding balance globally and broken down by the current active `Organization`.
5.  **Accounts Receivable Report (Saldos):**
    * Displays a list of clients whose total balance is greater than 0.
    * Columns: Full Name, Total Amount Owed.
    * *Balance Calculation Formula:* Sum of all `Charge` transactions minus the sum of all `Payment` and `Void` transactions.
